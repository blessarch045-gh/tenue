```
コメント案2:（季節・イベント系投稿への反応パターン）

- 営業感を出さない
```

```
以下の手順で週次リサーチを実行してください。

## 0. プロジェクト状態の把握
- .project/status.md を読む
- .project/weekly-goals.md を読む
- .project/pending.md を読む
- 上記を踏まえて以降のタスクを実行する

## 1. ドキュメント読み込み
- docs/マーケティング.md を読む
- docs/ブランド.md を読む
- docs/research/ 配下の直近レポートを確認し、
  前回との重複を避ける

## 2. Instagramトレンドリサーチ
以下の観点で直近1週間のファッション系投稿トレンドを調査する：
- 伸びているハッシュタグ（コーデ記録・着回し・季節系）
- 人気の投稿フォーマット（カルーセル/リール/静止画）
- エンゲージメントが高い投稿の共通点（構図・文体・時間帯）
- 今の季節・時期に合ったテーマの傾向

調査結果を以下の形式でまとめる：

### 注目ハッシュタグ（5個）
タグ名 / 推定投稿数の傾向 / tenue での活用案

### 投稿フォーマットの傾向
今週特に伸びている形式と、その理由の分析

### エンゲージメントの高い投稿パターン（3件）
投稿の特徴 / なぜ伸びたかの分析 / tenue の SNS に活かせるポイント

## 3. 競合アプリリサーチ
「コーディネート記録」「ファッション管理」「クローゼット管理」
に該当するアプリの最新動向を幅広く調査する：
- 新規リリースされたアプリ
- 既存アプリの注目アップデート
- App Store でのランキング変動やレビュー傾向
- ユーザーの不満点（tenue の差別化ヒントになる）

調査結果を以下の形式でまとめる：

### 今週の注目アプリ（最大3つ）
アプリ名 / 概要 / 注目ポイント / tenue との差別化の視点

### ユーザーの声から見える機会
競合アプリのレビューやSNSの声から、tenue が応えられそうなニーズ

## 4. tenue への示唆

### 4-1. SNS運用への反映（最優先）
今週の調査を踏まえて、来週のSNS運用に即座に反映すべき点：
- 採用すべきハッシュタグの入れ替え提案
- 投稿フォーマットの調整提案（リール増やす等）
- 投稿時間帯の調整提案
- キャプションのトーンや切り口の提案

### 4-2. App Store 掲載文・ASO への示唆
ユーザーが使っている検索キーワードや、
競合アプリのストア掲載文から得られる示唆

### 4-3. アプリ機能への示唆
競合のアップデートやユーザーの不満から見える機能のヒント

### 4-4. ブランドポジショニング
競合との差別化ポイントの再確認・新たな発見
```

## ステップ7: 枠3「週次ヘルスチェック」（週次・月曜実行）

### 設定場所
https://claude.ai/code/scheduled/new

### 設定内容
- **名前**: 週次プロジェクトヘルスチェック
- **リポジトリ**: tenue
- **頻度**: 週次（月曜）
- **プロンプト**: 以下をコピペ

```
以下の手順で tenue リポジトリの週次ヘルスチェックを実行してください。

## 0. プロジェクト状態の把握
- .project/status.md を読む
- .project/weekly-goals.md を読む
- .project/pending.md を読む
- 上記を踏まえて以降のタスクを実行する

## 1. ドキュメント整合性チェック

### 1-1. 技術設計.md の検証
- docs/技術設計.md を読む
- 以下の項目が実際のコードと一致しているか検証する：
  - DB テーブル定義（profiles, coord_records）
  - Storage のパス規則・圧縮設定
  - 画面構成と画面ファイルの対応
  - ファイル配置ルールと実際のディレクトリ構造
- 乖離があれば具体的な差分を記録する

### 1-2. CLAUDE.md の検証
- CLAUDE.md を読む
- 以下を確認する：
  - 「現在のフェーズ」が実態と合っているか
  - 「参照ドキュメント」のファイルパスが全て有効か
  - 「作業ルール」「コードスタイル」が実際のコードと矛盾していないか

### 1-3. README.md の検証
- README.md を読む
- セットアップ手順、機能一覧が現在のコードと一致しているか確認

## 2. 依存関係チェック

### 2-1. セキュリティ監査
- npm audit を実行する
- 脆弱性がある場合：
  - 深刻度（critical/high/moderate/low）を分類
  - 影響範囲と対処方法を記載

### 2-2. パッケージ鮮度チェック
- npm outdated を実行する
- メジャーアップデートがあるパッケージを列挙
- Expo SDK や React Native に関わる重要な更新は特に明記

## 3. コード品質チェック
- TypeScript の型エラーがないか確認（npx tsc --noEmit）
- 未使用の import や変数がないか確認

## 4. レポート作成
以下の形式でまとめる：

### 今週のステータス: 🟢 問題なし / 🟡 要確認あり / 🔴 要対応あり

### ドキュメント乖離（あれば）
| ファイル | 項目 | 現状 | あるべき状態 |

### セキュリティ（あれば）
| パッケージ | 深刻度 | 内容 | 推奨対応 |

### パッケージ更新（重要なもののみ）
| パッケージ | 現在 | 最新 | 影響 |

### 型エラー（あれば）
エラー内容と該当ファイル

## 5. 対応

### 自動修正する（PRを作成してよい）
以下に該当する問題はPRで修正すること：
- docs/技術設計.md の記述がコードと一致していない → コードに合わせてドキュメントを修正
- CLAUDE.md のファイルパスが無効、またはフェーズが古い → 現状に合わせて修正
- README.md の機能一覧やセットアップ手順が古い → 現状に合わせて修正
- npm audit でパッチバージョンで修正可能な脆弱性 → パッチバージョンにアップデート

PRタイトル: 「🏥 自動修正: ヘルスチェック YYYY-MM-DD」

### Issueに留める（PRを作成しない）
以下に該当する問題はIssueのみ作成すること：
- メジャーバージョンのパッケージ更新
- TypeScript の型エラー（コード変更が必要）
- ブランド方針や作業ルールに関わる判断が必要な変更
- 修正の影響範囲が大きく、テストが必要なもの

Issueタイトル: 「⚠️ 要対応: [問題の概要]」

### 問題がない場合
何もしない。PRもIssueも作成しない。

## 6. 状態更新
- .project/log.md に今回の作業内容を追記する
- .project/status.md の技術状態を更新する
- Bless の判断が必要な事項があれば .project/pending.md に追記する
- 上記の変更も PR に含める
```
